# Learn 탭 정확성 감사 절차

세 층위로 감사한다. 1층은 자동화되어 있고, 2·3층은 수동/에이전트 검토다.

## 1층 — 자동 감사 (필수, 가장 먼저)

```bash
pnpm --filter @algo-gym/web test
```

`apps/web/scripts/verify-learn.ts`가 전 알고리즘을 무작위 입력 40회씩 실행해 독립 레퍼런스 구현과 대조한다. 실패 메시지에 알고리즘 id와 기대값/실제값이 포함된다. 레지스트리에 있는데 검증 루틴이 없는 알고리즘도 실패 처리된다.

## 2층 — 브라우저 동작 점검

```bash
pnpm --filter @algo-gym/web dev
```

전 페이지 자동 순회가 필요하면 Playwright로: 카탈로그 카드 수 = 레지스트리 수, 각 `/learn/<id>` 페이지에서 슬라이더 끝 이동·스텝·셔플 클릭, `pageerror`/콘솔 에러 수집. (이전 세션에서 쓴 패턴: chromium headless, `algorithms`를 직접 import해 순회)

## 3층 — 교육 내용 사실 검증

자동화가 못 잡는 부분이므로 알고리즘 파일을 직접 읽으며 확인한다:

- **표시 코드**: `code` 배열을 학습자가 복사하면 실제로 동작하는가? 시뮬레이션과 같은 알고리즘인가?
- **라인 참조**: `step({ line })`이 가리키는 줄이 메시지 내용과 일치하는가?
- **복잡도 주장**: `complexity` 필드와 insight의 O(...) 주장이 정확한가? (예: 퀵 정렬 최악 O(n²), 유니온 파인드 α(n), 모노토닉 스택의 분할 상환 O(n))
- **불변식·정리 주장**: "BFS는 최단 경로 보장(균일 가중치 전제)", "다익스트라는 음수 간선 불가" 같은 전제 조건이 빠지지 않았는가?
- **LeetCode 문제 번호**: insight에 적힌 번호가 실제 그 문제인가?

대규모 검토는 general-purpose 에이전트에 위 체크리스트를 주고 위임하는 것이 효율적이다.

## 발견 시 처리

- 시뮬레이션 버그: 수정 후 1층 재실행. 해당 버그를 잡는 검증이 없었다면 verify-learn.ts에 검증을 보강한다.
- 교육 내용 오류: 메시지/insight 수정. 수치 주장은 근거를 확인하고 고친다.
- 수정 후 `pnpm lint && pnpm test` 전체 통과를 확인하고 커밋한다.
